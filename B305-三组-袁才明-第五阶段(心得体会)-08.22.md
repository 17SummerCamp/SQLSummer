# MySQL心得体会
##### 语句的构造
```
	自从PDF上出现子句(clause)之后心中就已暗叫不好，看到INSERT INTO···SELECT便证明是事实了，常用到的基础的语句并不多，但是为了应对多种情况，不得不用心的使用子句，虽然可以每步完成一小部分功能，但是不免使得查询文件过于冗长。刚开始学习C语言的时候，总是希望自己的程序越短越好，像是在线提交平台也会计算代码长度，自然也就生出了孰优孰劣的心思。通过学习，我知道了MySQL的子句可以组合成完成复杂功能的语句，其实道理也就像一些编程语言一样，我们学习稍微简单的知识，最后完成较为复杂的工作，学习正是如此，牢固的掌握好“简单”的知识，最后再能漂亮地完成复杂的工作。它让我看到了组合的奇妙，更让我看到了“简单”的不简单。
```
##### 关键字的使用
```
    不同于规则般的编程语言的关键字，SQL的关键字更像是可以随手拈来用工具。我认为有时使用不同的关键字有时也同样可以完成工作，但是这其中的效率、简介度、体验等等是完全不同的，虽然现在我并不怎么了解有关效率的问题，但是仅通过使用就足以发现端倪，所以我认为好关键字使用尤为重要。
    对于IN、ANY、HAVING等等，刚开始学习的时候并未深入了解，仅仅是知道对于某些特定的情况，使用这些关键字可以极大地简化工作量，因此而导致的问题也层出不穷，更有甚者或许只是放置的位置问题。
    有时候也会惊讶SQL对于处理速度的严格要求，像是使用删除时操作时DELETE与TRUNCATE TABLE谁更快些，当然，这些对于现在的我来说，影响并不大，但是日后相必定是要考虑这些 个问题。
```
##### 自以为是
```
    为什么这门语言叫做结构化查询语言呢？为什么创建的查询文件可以做任何事情？有些问题无关乎本质，并且乍一看也没有什么重要性可究，于是便私自作答（自以为是），现在想想或许当初就应刨根问底，或是说充满好奇心才能走得更远？对于名称的理解与对于知识的理解有没有促进作用呢？我认为现在就应是回答这些问题的时候了。一听到“查询”就应当想到这庞大的数据库，但是这样又不免陷入泥潭，即：主动思考与自以为是的界线何在？如何加以区分？考虑到人的精力有限，什么东西是我们应当“得过且过”的？，什么又是我们应当“刨根问底”的呢？但就这以“用途”来命名，我想当是应当且有必要了解的。
    因为语句的组件在我看来相当的复杂，想要做到一下子全部知道对于记忆来说无疑不太现实，于是便时常自己构造一些句子，能运行自然皆大欢喜，若是不能也不过尝试而已，最后倒也真解决了几个问题，由于存在赌博的成分，所以细想来若是不长记性，下次免不得又要尝试，最好是应是把自己摸索出来的记下来，日后若是能用到便是更好了。
```
